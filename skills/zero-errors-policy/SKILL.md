# Zero Errors Policy

## Core Principle

**The codebase is either clean or it is not. There is no "pre-existing" category.**

A task is not complete until lint and tests are zero-error — not "the same errors as before", not "no new errors I introduced", not "errors only in files I didn't touch". Zero errors. Full stop.

---

## The Two Checkpoints

### Checkpoint 1 — Before you write a single line

Run the full lint and test suite and record the result.

```bash
# Tests
bunx vitest run         # or: npm test / cargo test / pytest / go test ./...

# Lint / type-check
bunx eslint src         # or: bunx tsc --noEmit / cargo clippy / ruff check
```

**If errors exist before you start:**

- Fixing them IS your first task, before any feature work begins
- Do not proceed to implementation until baseline is clean
- If errors are in files you were not asked to touch, you still fix them
- "They were there when I arrived" is not a reason to leave them there

**If baseline is clean:** proceed to implementation.

### Checkpoint 2 — Before you mark the task complete

Run the same suite again. Required output: zero errors.

```bash
bunx vitest run
bunx eslint src
```

If errors exist: fix them. Do not commit, do not mark done, do not move to the next task.

---

## Forbidden Rationalizations

These thoughts mean STOP — you are rationalizing away a quality violation:

| Rationalization | Reality |
|-----------------|---------|
| "Zero new errors introduced by my changes" | New vs. pre-existing is irrelevant. The count must be zero. |
| "The same N errors as before my changes" | N errors is not zero. Fix them. |
| "These errors are in files I didn't touch" | You are responsible for the state of the whole codebase, not just your diff. |
| "These are pre-existing and unrelated to my task" | All errors are your responsibility. Fix them or the task is not done. |
| "This is consistent with how the rest of the codebase handles it" | A widespread bad pattern is still bad. Fix it. |
| "The task only asked me to add X, not fix linting" | Every task implicitly requires leaving the codebase clean. |
| "These are environment errors (missing env vars, etc.)" | Real errors get fixed. Known false positives get documented in CLAUDE.md with a clear justification — never silently skipped. |
| "TypeScript errors are in stub/generated files" | Fix them, move them out of the compiler path, or add them to tsconfig `exclude` with a comment. |

---

## Red Flags — Stop and Fix

- You are about to write "these errors are pre-existing"
- You are about to write "no new errors introduced"
- You are counting errors instead of fixing them
- You ran lint or tests and skipped past failures
- You are comparing your diff to the error count instead of comparing the error count to zero

**All of these mean: fix the errors before proceeding.**

---

## When Baseline Errors Are Genuinely Out of Scope

This is rare. The only acceptable exception: errors that are structurally impossible to fix in the current task (e.g., a missing external service, a generated file that must be regenerated by a CI step).

**Required handling:**
1. Document in CLAUDE.md or a comment exactly why the error cannot be fixed now
2. Create a follow-up task for fixing it
3. Get explicit acknowledgment from the user that this exception is acceptable

Silent tolerance of errors is never acceptable. Documented, tracked exceptions with user sign-off are.

## Quick Reference

```
Before task:  run lint + tests → errors? → fix first → then implement
After task:   run lint + tests → errors? → fix → then mark complete
Verdict:      zero errors = done / any errors = not done
```

---

## Common Mistakes

**"I only touched two files, so errors in other files aren't my problem"**
You share a codebase. Clean it or flag it. Never ignore it.

**Running tests but not lint (or vice versa)**
Both checkpoints require both checks. A passing test suite with lint errors is not clean.

**Checking a subset of tests**
Run the full suite. A failing test you didn't run is still a failing test.
