# Semantic Styles — Rokkit

Rokkit's token pipeline has three layers. Understanding all three prevents the common
failure of mixing raw color values into components and losing the consistency guarantee.

```
rokkit.config.js
  palettes + skin + colorSpace
       ↓
  UnoCSS preset (presetRokkit)
  generates CSS custom properties + utility classes
       ↓
  Components use utility classes (bg-surface-z1, text-primary-z5)
  or CSS vars (--color-surface-z0) for CSS-only contexts
```

---

## Layer 1 — Palettes

**What:** A palette is a named 11-stop color scale from shade `50` (lightest) to shade
`950` (darkest). You provide it in `rokkit.config.js` under `palettes:`.

**Why:** Rokkit needs a range of shades for each role so that hover, active, text-on-fill,
and fill-on-surface states all come from the same color family. 11 stops gives enough
resolution to express all these relationships without banding.

**Two formats — RGB hex vs OKLCH bare components:**

*RGB hex* — standard web colors, immediate tool support, used when you have existing palette:
```js
palettes: {
  shu: { 50: '#FEF3EE', 500: '#E8552B', 950: '#3C100A' }
}
```
Set `colorSpace: 'rgb'` (or omit — rgb is the default). CSS vars are stored as
comma-separated `R,G,B` triplets: `--color-primary-500: 232,85,43`. Used in components
as `rgb(var(--color-primary-500) / 0.5)` for alpha support.

*OKLCH bare components* — perceptual color space, uniform lightness steps, better for
programmatic palette generation and smooth gradients:
```js
palettes: {
  shu: { 50: '0.970 0.020 35', 500: '0.580 0.150 35', 950: '0.220 0.060 35' }
},
colorSpace: 'oklch'
```
CSS vars are stored as bare `L C H` components: `--color-primary-500: 0.580 0.150 35`.
Used as `oklch(var(--color-primary-500) / 0.5)`. This is the preferred format for
new custom palettes because the lightness axis is perceptually uniform — z-steps
look equally spaced to the human eye.

**When to use each:** Use RGB hex when you have an existing brand palette from a
designer or color tool. Use OKLCH when you are designing a palette from scratch —
increment L by equal steps and the result will look balanced without manual tuning.

## Layer 2 — Skin and skins

**What:** A skin maps semantic role names (surface, primary, success…) to palette names.
The mapping tells Rokkit which palette to use for each role.

**Why:** Decoupling role from palette name means you can swap the entire color scheme
by changing config, not component code. A component that uses `bg-surface-z1` does not
know whether `surface` maps to `sumi`, `slate`, or `zinc` — it just knows "the neutral
background at level 1."

### Single skin (`skin:`) — app has one fixed colormap

```js
skin: {
  surface:   'sumi',      // neutral background
  primary:   'shu',       // accent
  secondary: 'murasaki',  // secondary accent
  success:   'jade',
  warning:   'amber',
  danger:    'beni',
  error:     'beni',
  info:      'ai',
}
```

Use `skin:` when the app has no theme-switching requirement. The colormap is baked in.
`colors:` is accepted as a backward-compatible alias but prefer `skin:`.

### Multi-skin (`skins:`) — app supports named colormaps for switching

```js
skins: {
  default: { surface: 'sumi', primary: 'shu', secondary: 'murasaki' },
  ocean:   { surface: 'slate', primary: 'sky', secondary: 'teal' },
  violet:  { surface: 'zinc',  primary: 'violet', secondary: 'purple' },
}
```

`skins.default` is the active colormap on first load. The UI switches by toggling a
CSS class that loads a different `skin-*` shortcut generated by the preset.

**Rule — use `skin:` or `skins:`, not both.** They are mutually exclusive. Providing
`skins:` makes `skin:` ignored. If you only need one colormap, use `skin:`.

## Layer 2b — Dual-palette dark mode

**What:** A skin role can have a `{ light, dark }` object instead of a string.
Rokkit then generates two separate CSS var blocks — one for `:root` and one for
`[data-mode="dark"]`.

**Why:** Some colors that work on a light surface fail on a dark surface. A
warm-grey scale (`kami`) works for light-mode paper tones but its middle shades
look wrong as dark-mode backgrounds. A separate dark-optimised scale (`sumi`)
gives full control over both ends without compromise.

```js
skin: {
  surface: { light: 'kami', dark: 'sumi' },
  primary: 'shu',   // single palette — works in both modes
}
```

Generated output:
```css
:root {
  --color-surface-50: <kami-50 value>;
  --color-surface-500: <kami-500 value>;
  /* … */
}
[data-mode="dark"] {
  --color-surface-50: <sumi-50 value>;
  --color-surface-500: <sumi-500 value>;
  /* … */
}
```

**How dark mode flips:** `@rokkit/themes/base.css` maps z-tokens to shade numbers
with an inversion for dark mode:
- Light: `z0 → shade-50` (lightest), `z9 → shade-900` (darkest)
- Dark: `z0 → shade-950` (darkest surface becomes background), `z9 → shade-100`
  (lightest becomes foreground text)

This means `bg-surface-z0` is automatically the right background in both modes —
no component-level dark mode code needed.

**When to use:** Use dual-palette only when a single scale cannot serve both light and
dark well. The `kami`/`sumi` split is the canonical example. For most accent colors
(`primary`, `success`, `warning`) a single palette adapts fine through the z-flip.

## Layer 3 — The z-scale: CSS variables and utility classes

This is the layer components actually use.

### How z-maps to shades

| z level | Light mode shade | Dark mode shade | Semantic meaning |
|---------|-----------------|-----------------|-----------------|
| z0 | 50 | 950 | App background / deepest ink |
| z1 | 100 | 900 | Card background / very dark |
| z2 | 200 | 800 | Elevated surface / dark |
| z3 | 300 | 700 | Hover fill / medium-dark |
| z4 | 400 | 600 | Subtle fill / medium |
| z5 | 500 | 500 | Mid-point (same both modes) |
| z6 | 600 | 400 | Strong border / muted fill |
| z7 | 700 | 300 | Secondary text |
| z8 | 800 | 200 | Sub-heading text |
| z9 | 900 | 100 | Primary text / brightest paper |
| z10 | 950 | 50 | Deep shadow / pure paper |

**Why this matters:** When you use `text-surface-z9`, you get dark text in light mode
and light text in dark mode automatically. You never write `dark:text-white` or
`dark:text-surface-100` — the z-flip handles it.

### CSS custom properties

Rokkit generates `--color-{role}-{shade}` vars (per palette) and `--color-{role}-z{n}`
shortcut vars (per z-level):

```css
/* Available for CSS-only contexts (animations, pseudo-elements, color-mix) */
--color-surface-z0      /* app background */
--color-surface-z1      /* card background */
--color-surface-z9      /* primary text */
--color-primary-z5      /* mid accent */
--color-success-z1      /* success tint bg */
```

**Usage in CSS:**
```css
/* With colorSpace: 'rgb' */
background: rgb(var(--color-surface-z0) / 0.9);

/* With colorSpace: 'oklch' */
background: oklch(var(--color-surface-z0) / 0.9);
```

**Critical:** Do not use `background-color: var(--color-surface-z0)` without wrapping
in `rgb()` or `oklch()` — the var contains bare components, not a complete color value.

### Utility class reference

**Background:**
```
bg-surface-z0    app background
bg-surface-z1    card, panel
bg-surface-z2    elevated card, hover target
bg-surface-z3    active/pressed fill
bg-primary-z1    primary tint (very light badge bg)
bg-primary-z5    primary fill (button, chip)
bg-success-z1    success tint
bg-warning-z1    warning tint
bg-danger-z1     danger tint
```

**Text:**
```
text-surface-z9  primary text (headings, body)
text-surface-z7  secondary text (captions, descriptions)
text-surface-z5  muted text (placeholders, disabled)
text-surface-z4  faint text (timestamps, secondary metadata)
text-primary-z5  accent text (links, active labels)
text-primary-z6  stronger accent (hover state of links)
text-success-z6  success text
text-warning-z6  warning text
text-danger-z6   error text
```

**Border:**
```
border border-surface-z2    hairline (cards, dividers)
border border-surface-z3    standard border (inputs, panels)
border border-surface-z4    strong border (active card, selected)
border border-primary-z4    primary focus ring fill
```

**Common patterns:**

*Card:*
```html
<div class="bg-surface-z1 border border-surface-z2 rounded-md p-4">
  <h3 class="text-surface-z9 text-sm font-medium">Title</h3>
  <p class="text-surface-z6 text-xs mt-1">Description</p>
</div>
```

*Status badge:*
```html
<span class="bg-success-z1 text-success-z6 text-xs px-2 py-0.5 rounded-full">Active</span>
```

*Input:*
```html
<input class="bg-surface-z0 border border-surface-z3 rounded-md px-3 py-2 text-surface-z9
              focus:border-primary-z4 focus:outline-none text-[13px]" />
```

## Typography configuration

**What:** Set `--font-sans`, `--font-mono`, `--font-heading` via `rokkit.config.js`.
The preset generates CSS vars and maps them to Tailwind's `fontFamily.body`,
`fontFamily.mono`, `fontFamily.heading` theme keys.

**Why:** Font-family should not be hard-coded in components. Centralising in config
means a single line change swaps the typeface across the entire app.

```js
typography: {
  sans:    "'Inter Variable', 'Inter', system-ui, sans-serif",
  mono:    "'JetBrains Mono', 'SF Mono', Menlo, monospace",
  heading: "'Fraunces', 'Iowan Old Style', Georgia, serif",
}
```

Generated CSS:
```css
:root {
  --font-sans: 'Inter Variable', 'Inter', system-ui, sans-serif;
  --font-mono: 'JetBrains Mono', 'SF Mono', Menlo, monospace;
  --font-heading: 'Fraunces', 'Iowan Old Style', Georgia, serif;
}
```

**Usage in components:**
```html
<!-- body text (sans is mapped to font-body in Tailwind theme) -->
<p class="font-body text-[13px] text-surface-z7">Description</p>

<!-- monospace code/paths -->
<code class="font-mono text-xs text-surface-z6">path/to/file.ts</code>

<!-- display heading -->
<h1 class="font-heading text-2xl text-surface-z9">Page Title</h1>
```

## Shape / radius configuration

**What:** `shape: { radius: 'soft' }` selects a named preset that generates
`--radius-sm`, `--radius-md`, `--radius-lg`, `--radius-xl`, `--radius-full`
CSS custom properties.

**Why:** Consistent curvature is part of the design language. When components pick
their own `border-radius` values ad hoc, the UI reads as incoherent even when
individual screens look fine. Centralising radius means adjusting `'soft'` to
`'rounded'` in config instantly shifts the personality of the whole app.

**Named presets:**

| Preset | sm | md | lg | xl | full | Personality |
|--------|----|----|----|----|------|-------------|
| `sharp` | 0 | 0 | 0 | 0 | 9999px | Minimal, editorial |
| `soft` | 2px | 6px | 10px | 12px | 9999px | Professional, calm |
| `rounded` | 4px | 8px | 12px | 16px | 9999px | Friendly, modern |
| `pill` | 9999px | 9999px | 9999px | 9999px | 9999px | Playful, pill-native |

**Custom preset:** Pass an object for full control:
```js
shape: { radius: { sm: '2px', md: '6px', lg: '10px', xl: '14px', full: '9999px' } }
```

**Usage in components — use Tailwind radius utilities that map to the vars:**
```html
<div class="rounded-md">    <!-- uses --radius-md -->
<span class="rounded-sm">   <!-- uses --radius-sm -->
<button class="rounded-lg"> <!-- uses --radius-lg -->
<div class="rounded-full">  <!-- uses --radius-full (pill/avatar) -->
```

**Why not `rounded-[6px]`?** Arbitrary values bypass the token system — they will
not change when you update `shape.radius` in config. Always use the named Tailwind
size that maps to the CSS var.

## Spacing in Rokkit apps

Rokkit does not have a proprietary spacing token system — it inherits Tailwind's
4px-grid spacing. The key rule: **stay on the 4px grid, use Tailwind names.**

**Grid reference:**

| Tailwind | px | rem | When |
|----------|----|-----|------|
| `1` | 4px | 0.25rem | Tight inline gaps |
| `1.5` | 6px | 0.375rem | Sub-grid step |
| `2` | 8px | 0.5rem | Component internal gap |
| `2.5` | 10px | 0.625rem | Sub-grid step |
| `3` | 12px | 0.75rem | Standard gap, compact padding |
| `3.5` | 14px | 0.875rem | Sub-grid step |
| `4` | 16px | 1rem | Default padding, comfortable gap |
| `5` | 20px | 1.25rem | Section gap |
| `6` | 24px | 1.5rem | Panel padding |
| `8` | 32px | 2rem | Section separation |
| `10` | 40px | 2.5rem | Large whitespace |
| `12` | 48px | 3rem | Page-level padding |
| `16` | 64px | 4rem | Hero / stage whitespace |

**Screen density:**
- **Compact** (data tables, sidebars): base 4–8px, prefer `gap-1`/`gap-2`, `p-2`/`p-3`
- **Comfortable** (cards, forms): base 8–16px, prefer `gap-3`/`gap-4`, `p-4`/`p-6`
- **Relaxed** (marketing, landing pages): base 16–24px+

## Full config example

```js
// rokkit.config.js
export default {
  palettes: {
    kami: {  // light surface — warm paper tones
      50: '0.985 0.005 85', 100: '0.975 0.008 85', /* … */ 950: '0.170 0.010 50'
    },
    sumi: {  // dark surface — ink tones
      50: '0.975 0.008 85', /* … */ 950: '0.170 0.010 50'
    },
    shu: {   // vermillion accent
      50: '0.970 0.020 35', 500: '0.580 0.150 35', 950: '0.220 0.060 35'
    },
  },

  colorSpace: 'oklch',  // required when palette values are bare L C H components

  skin: {
    surface:   { light: 'kami', dark: 'sumi' }, // dual-palette: different scales per mode
    primary:   'shu',      // single palette — z-flip handles both modes
    success:   'jade',
    warning:   'kohaku',
    danger:    'shu',
  },

  typography: {
    sans:    "'Inter Variable', system-ui, sans-serif",
    mono:    "'JetBrains Mono', monospace",
    heading: "'Fraunces', Georgia, serif",
  },

  shape: { radius: 'soft' },

  switcher:   'manual',     // 'system' | 'manual' | 'full'
  storageKey: 'app-theme',
};
```

## Common mistakes

| Mistake | Why it fails | Fix |
|---|---|---|
| `color: #3D3730` in a component | Breaks on theme change; opaque meaning | `text-surface-z8` |
| `background: var(--color-surface-z0)` | Bare component — not a valid color | `rgb(var(--color-surface-z0) / 1)` or use `bg-surface-z0` |
| Using `skin:` and `skins:` together | `skins:` silently overrides `skin:` | Pick one |
| `border-radius: 6px` hardcoded | Bypasses shape config | `rounded-md` |
| `font-family: 'Inter'` in component | Bypasses typography config | `font-body` (Tailwind class) |
| `oklch(0.58 0.15 35)` in component CSS | Loses opacity composability | Use `oklch(var(--color-primary-z5) / alpha)` |
| Mixing `colorSpace: 'oklch'` with hex palettes | CSS vars will be bare `0.58 0.15 35` but wrapping will be `rgb(...)` | Match colorSpace to palette format |
| `@apply bg-surface-z0` on `body {}` | UnoCSS generates a descendant combinator that can't match `body` when `data-mode` is on `body` | Use `body { background: oklch(var(--color-surface-z0) / 1); }` directly |
