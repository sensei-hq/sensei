# Phase 4: Multi-Agent Support ‚Äî Design Spec

## Goal

`sensei setup --agent claude` generates four project-specific skill files and installs them into `~/.claude/skills/`. Skills are generated by Claude (via `ClaudeBackend`) from a `ProjectProfile` derived from the indexed codebase, then validated by Claude before writing. Skill status is visible in a dashboard Agent Skills view.

---

## Problem

Sensei already indexes the codebase and tracks sessions, but agents starting work on a repo have no project-specific skills to orient them. Skills written by hand are generic. Phase 4 generates project-specific skills automatically ‚Äî actual package names, actual CLI commands, actual patterns ‚Äî and validates them so agents get accurate guidance on first load.

---

## Architecture

```
sensei setup --agent claude
  ‚Üí extractProjectProfile(db, repoId, repoPath)    reads symbols + config
  ‚Üí ClaudeBackend.generate()                        4 prompts, one per category
  ‚Üí SkillValidator.validate()                       Claude reviews each for accuracy
  ‚Üí ClaudeAdapter.writeSkills()                     writes to ~/.claude/skills/
  ‚Üí .sensei/agent-skills.json                       metadata (paths, timestamps)

install_skills MCP tool
  ‚Üí same pipeline, triggered from active agent session

Dashboard /repos/[id]/agents
  ‚Üí reads .sensei/agent-skills.json                 skill status (present/stale/missing)
  ‚Üí Regenerate button                               form action spawns sensei setup --agent claude
```

Skill metadata stored in `.sensei/agent-skills.json` ‚Äî no new DB migration required.

## Types (`packages/shared/src/types.ts`)

```typescript
export interface ProjectProfile {
  repoName: string;
  repoPath: string;
  dominantLanguage: string;             // 'typescript' | 'python' | etc.
  framework: string | null;             // 'sveltekit' | 'react' | 'express' | null
  packageNames: string[];               // monorepo packages e.g. ['engine', 'cli', 'shared']
  keySymbols: string[];                 // top 20 most-referenced exported symbols
  testPattern: string;                  // e.g. '*.spec.ts'
  cliCommands: Record<string, string>;  // from package.json scripts
  senseiConfig: string;                 // serialised .sensei/config.yaml summary
}

export interface AgentSkillFile {
  category: 'orientation' | 'workflow' | 'context' | 'patterns';
  path: string;          // absolute path to written skill file
  generatedAt: string;   // ISO timestamp
}

export interface AgentSkillsManifest {
  agent: 'claude';
  repoSlug: string;
  skills: AgentSkillFile[];
  updatedAt: string;
}
```

### `extractProjectProfile`

```typescript
export async function extractProjectProfile(
  db: SupabaseClient,
  repoId: string,
  repoPath: string
): Promise<ProjectProfile>
```

- Queries `repos` table for `name`
- Queries `symbols` for top 20 by reference count (or alphabetical fallback), and file extension distribution for `dominantLanguage`
- Reads `package.json` from `repoPath`: extracts `scripts` ‚Üí `cliCommands`, `dependencies`+`devDependencies` ‚Üí `framework` detection, `workspaces` ‚Üí `packageNames`
- Reads `.sensei/config.yaml` for `senseiConfig` summary
- Throws on DB error; throws if `package.json` missing

### `SkillGenerator`

```typescript
export class SkillGenerator {
  constructor(
    private model: ModelBackend,
    private profile: ProjectProfile,
    private validator: SkillValidator  // injected ‚Äî not constructed internally
  ) {}

  async generate(): Promise<Record<'orientation' | 'workflow' | 'context' | 'patterns', string>>
  // For each category: generate ‚Üí validate ‚Üí retry up to 2x on invalid result
  // Throws if still invalid after 2 retries, listing the remaining issues
}
```

Four prompts, each injecting relevant `ProjectProfile` fields:

| Category | Profile fields used |
|----------|-------------------|
| `orientation` | repoName, packageNames, keySymbols, dominantLanguage, framework |
| `workflow` | cliCommands, testPattern, senseiConfig |
| `context` | keySymbols, packageNames, senseiConfig |
| `patterns` | dominantLanguage, framework, testPattern, cliCommands |

Each prompt instructs Claude to write a valid skill in SKILL.md format with correct YAML frontmatter.

### `SkillValidator`

```typescript
export class SkillValidator {
  constructor(private model: ModelBackend, private profile: ProjectProfile) {}

  async validate(category: string, skillMarkdown: string): Promise<{ valid: boolean; issues: string[] }>
}
```

Sends review prompt: *"Check this skill for accuracy against the project profile. Verify: package names match, commands are real, no hallucinated file paths. Reply VALID or list issues."*

`SkillValidator` is constructed separately and injected into `SkillGenerator`. This keeps the retry loop in `SkillGenerator.generate()` and the validation logic in `SkillValidator.validate()` independently testable.

### `AgentAdapter` interface

```typescript
export interface AgentAdapter {
  skillsDir: string;
  writeSkills(
    skills: Record<string, string>,
    repoSlug: string
  ): Promise<AgentSkillFile[]>;
  installedSkills(repoSlug: string): Promise<AgentSkillFile[]>;
}
```

### `ClaudeAdapter`

- `skillsDir`: resolved at construction time via `path.join(os.homedir(), '.claude', 'skills')` ‚Äî never uses `~` literally (Node.js `fs` does not expand it)
- `writeSkills`: writes `sensei-{repoSlug}-{category}.md` for each category; creates dir if missing; returns `AgentSkillFile[]` with `generatedAt: new Date().toISOString()`
- `installedSkills`: reads dir, filters by `sensei-{repoSlug}-` prefix, returns list with mtime as `generatedAt`

Note: `ClaudeAdapter` is placed in `packages/engine/src/agent/` alongside the `AgentAdapter` interface. Engine is permitted I/O side effects at the adapter layer (consistent with `Indexer` writing to Supabase and `Scanner` reading from disk).

## CLI Changes (`packages/cli/src/commands/setup.ts`)

New `setupAgent(repoPath: string, agent: 'claude'): Promise<void>` function:

```
1. intro("sensei setup --agent claude")
2. makeSenseiClient(repoPath) + loadSenseiConfig(repoPath) ‚Äî throws if not initialised
3. extractProjectProfile(db, repoId, repoPath)
4. new ClaudeBackend() ‚Üí backend.init() ‚Äî fails fast if ANTHROPIC_API_KEY missing
5. new SkillGenerator(backend, profile).generate()
   ‚Äî spinner: "Generating {category} skill..." √ó 4
6. new SkillValidator(backend, profile).validate() per skill (within generator loop)
7. new ClaudeAdapter().writeSkills(skills, repoSlug)
8. write .sensei/agent-skills.json (AgentSkillsManifest)
9. outro: "4 skill files written to ~/.claude/skills/"
```

`cli.ts` gains `--agent <agent>` option on the `setup` subcommand. Accepted value: `claude`. Unsupported values print "Agent '<value>' is not yet supported. Supported: claude" and exit with code 1.

## MCP Tool (`packages/server/src/tools/install-skills.ts`)

```typescript
export async function installSkillsTool(
  db: SupabaseClient,
  repoId: string,
  repoPath: string
): Promise<{ filesWritten: string[]; errors: string[] }>
```

`installSkillsTool` runs the full pipeline including writing `.sensei/agent-skills.json` (same as `setupAgent`) ‚Äî the dashboard reads this manifest, so it must be updated by both entry points. `repoPath` is available in the MCP server closure via `opts.repoPath`.

Registered in `mcp-server.ts`:

```typescript
server.tool("install_skills", {}, async () => {
  const client = await getClient();
  const result = await installSkillsTool(client as any, repoId, opts.repoPath);
  beat(client, "install_skills", result.errors.length === 0);
  return { content: [{ type: "text", text: JSON.stringify(result) }] };
});
```

## Error Handling

| Operation | On error |
|-----------|----------|
| `extractProjectProfile` | Throws ‚Äî setup fails clearly |
| `ClaudeBackend.init()` | Throws ‚Äî no `ANTHROPIC_API_KEY` message |
| `SkillGenerator.generate()` | Throws after 2 retries with validator issues listed |
| `ClaudeAdapter.writeSkills()` | Throws ‚Äî filesystem error surfaced |
| `installSkillsTool` | Returns `errors[]` ‚Äî never throws |
| Dashboard load | Returns empty state on missing manifest |

## Test Scope

**`project-profile.spec.ts`**:
- `extractProjectProfile` with fixture symbols returns correct `dominantLanguage` and `keySymbols`
- Missing `package.json` throws
- DB error (mocked Supabase returning error) throws

**`skill-generator.spec.ts`**:
- `generate()` with mock `ModelBackend` calls `generate()` exactly 4 times
- Retries on invalid validation result (up to 2 retries)
- Throws after 2 failed retries

**`skill-validator.spec.ts`**:
- Returns `{ valid: true }` when model returns "VALID"
- Returns `{ valid: false, issues: [...] }` when model returns issues list

**`claude-adapter.spec.ts`**:
- `writeSkills()` creates correct filenames in a temp dir
- `installedSkills()` returns only files matching the repo slug prefix

**`claude-backend.spec.ts`** (in `packages/server`):
- `init()` throws when `ANTHROPIC_API_KEY` absent from env
- `embed()` throws `NotImplementedError`

## Done When

- `sensei setup --agent claude` runs on the sensei repo ‚Üí 4 skill files written to `~/.claude/skills/` containing sensei-specific content (e.g. references to `packages/engine`, `bun run test`)
- Each skill validated by Claude before writing ‚Äî invalid skills cause a clear error with listed issues
- `install_skills` MCP tool produces the same result from within an agent session
- Dashboard `/repos/[id]/agents` shows all 4 skill files with status badges
- "Regenerate Skills" button re-runs the pipeline and updates the view
