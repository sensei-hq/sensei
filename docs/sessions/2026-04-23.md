## Learnings from dbd apply/import verification

Issues encountered and resolved during `dbd reset && dbd apply && dbd import`:

1. **Staging tables must use `DROP TABLE + CREATE TABLE`**, not `CREATE TABLE IF NOT EXISTS`. Otherwise schema changes (dropped columns) aren't picked up on re-apply.

2. **dbd JSONL import sets all columns explicitly** — missing keys become `null`, overriding `DEFAULT`. Make nullable columns in staging tables nullable (no `NOT NULL`), and use `COALESCE` in import procedures.

3. **All enums must live in one schema** (sensei). dbd applies enums before tables, but cross-schema enum references don't resolve unless the enum's schema is in the search_path. Consolidating to sensei simplifies everything.

4. **Import procedures need explicit enum casts** — `stg.capability::sensei.model_capability`, `stg.authentication_type::sensei.auth_type`, etc. PostgreSQL won't auto-cast text to enum.

5. **Cross-schema FK references work** if the referenced schema is in the `search_path`. Every schema's DDLs include their dependency schemas: `inference` includes `sensei`, `activity` includes `sensei`, `history` includes `sensei`.

6. **`CREATE TABLE IF NOT EXISTS` doesn't alter existing tables** — if you change a table (drop a column), you must `dbd reset` to recreate from scratch. No incremental migration yet.

7. **Inline `create type if not exists` in table DDLs** is needed when dbd might process files in alphabetical order and a type is defined in a later file. With dbd's enum support (separate `ddl/enum/` folder), this is no longer needed — enums are applied first.

### dbd workflow

```
dbd reset    — drop all schemas
dbd apply    — create schemas, enums, tables, views, functions, procedures
dbd import   — load JSONL seed data via staging tables + import procedures
dbd inspect  — verify referential integrity
```

All four pass cleanly as of this commit.
